# 技术文档改写示例

## 示例一：字段名不要裸奔

原文：

```markdown
用例中的 checks 字段只引用 check_id。
```

改写：

```markdown
用例里的“检查项”只引用已定义的检查项编号，也就是 `checks` 和 `check_id`。
```

原因：非研发读者先看到中文含义，研发读者也能拿到准确字段名。

## 示例二：保留必要工程术语

原文：

```markdown
幂等、回注、干跑和兜底都要处理。
```

改写：

```markdown
这段逻辑要支持重复执行不重复写入，也就是幂等；同时要能把结果写回原系统，支持只跑流程但不真实发送的干跑模式，并在异常时进入兜底处理。
```

原因：工程术语可以保留，但要补一句中文解释，让混合读者能理解约束。

## 示例三：把实现词换成可理解动作

原文：

```markdown
主流程负责结构化事实预处理、触发被测链路、出站截获和硬兜底。
```

改写：

```markdown
工具会先整理可核对的事实，触发被测流程，并在真实发送前拦截用户可见消息；遇到高风险动作时，直接拦截或交给人工复核。
```

原因：改写后读者能看见流程动作和风险处理方式。
