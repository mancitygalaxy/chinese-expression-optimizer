# 中文表达优化规则

这是一套给 AI 使用的中文表达规则。目标是让中文输出自然、清楚、顺口，少一点翻译腔、客服腔、营销腔和空泛黑话，同时保留事实、数字、路径、字段名、业务口径和风险边界。

这份规则和 Codex Skill `chinese-expression-optimizer` 使用同一套核心口径。你可以直接复制本文件，也可以安装仓库里的 Skill。

## 基本原则

- 普通概念优先用中文，例如“规则”“流程”“输入”“输出”“示例”“当前情况”“建议”。
- 专有名词、产品名、协议名、命令、路径、文件名、字段名、函数名和代码符号保留原文，并用反引号标出。
- 字段名要配中文说明，优先写成“中文说明 + `field_name`”，例如“检查项 `checks`”“预期结果 `expected`”。
- 把生硬英文句式改成自然中文，例如把“基于……进行……”“通过……来……”“被用于……处理……”改成更直接的动作句。
- 减少客服腔、营销腔、公文腔和空泛黑话；把抽象词改成能看见动作和结果的说法。
- 标题、结论句和规则句优先使用正向表达：先说是什么、怎么做、做到什么结果；需要划边界时，先给正向口径，再补边界说明。
- 面向业务、产品、测试、教研读者时，先讲动作和结果，再补必要术语。
- 面向研发读者时，保留必要工程术语和准确标识，并补一句中文解释，让术语服务理解。
- 改写时保持事实、数字、路径、业务口径、项目结论和风险边界一致；拿不准就标为“待确认”。

## 正向表达门禁

标题、模块名、结论句和规则句优先写成正向表达。

推荐写法：

- 先写“要做什么、怎么做、做到什么结果”。
- 需要划边界时，先给正向口径，再补边界说明。
- 把“不是做 A，而是做 B”改成“本方案做 B；A 作为历史背景或边界说明”。
- 把“不新增某文件”改成“首期文件范围固定为哪些文件”。
- 把“不要把 X 写进正文”改成“X 归入某某文档或补充说明”。

否定表达可以保留在必要的安全边界、禁止事项、踩坑纠偏和风险提示里；正文主干优先用正向动作句组织。

## Markdown 自检

创建、改写或整理 Markdown 文件后，检查新增或修改段落：

- 是否先给结论，再展开依据、步骤、规则、示例或风险。
- 长段落是否能拆成短段、列表或小标题。
- 中英文混用是否必要。
- 命令、路径、字段名、代码符号是否用反引号标出。
- 字段名是否有中文说明，例如“检查项 `checks`”。
- 表达是否自然，是否更像真实中文使用者在对应场景里会写的话。
- 标题、结论句和规则句是否以正向表达为主。
- 事实和口径是否保持一致。
- 最终回复是否标明 `中文表达自检：已做`；本轮未新增或修改 Markdown 时，标明 `中文表达自检：不适用`。
