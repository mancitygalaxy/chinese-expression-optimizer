# 中文表达优化

## 定位

优化中文表达，同时保留作者原意、事实、数字、业务口径、技术术语和文档结构，让表达更自然、更清楚，更符合目标读者的阅读习惯。

适合处理中文写作、中文 Markdown、工作文档、规则、方案、评审材料、交接卡、话术和汇报稿。

## 基本约束

- 保留事实、数字、名称、路径、命令、字段名、业务口径和已有决策。
- 缺失背景会影响改写判断时，简短追问；当前不宜追问时标为 `待确认`。
- 优先使用自然中文语序，把生硬英文句式改成直接的中文动作。
- 只有产品名、专有名词、协议名、命令、路径、代码符号、字段名和行业内固定说法保留英文。
- 命令、路径、文件名、字段名、代码符号和字面标识用反引号标出。
- 技术标识优先写成“中文说明 + `identifier`”，例如“检查项 `checks`”“预期结果 `expected`”。
- 保留原文有用的信息密度、工作痕迹、约束和风险，让表达更清楚，同时保持材料可执行。

## 工作流程

1. 判断任务模式：
   - **直接改写**：返回优化后的版本。
   - **只审校**：指出问题，并给出替换建议。
   - **Markdown 文件修改**：直接改文件，并复查新增或修改段落。
   - **批量审校**：先分组、定术语，再判断修改优先级。
2. 判断目标读者：
   - 业务、产品、运营、教研、测试、研发、管理层或公开读者。
   - 面向混合读者时，正文先用中文讲清意思，再把技术标识作为补充。
3. 分层处理：
   - 结构：先给结论，再展开依据、规则、步骤、风险或示例。
   - 句子：把生硬翻译腔改成直接的中文动作。
   - 术语：普通概念中文化，必要标识保留原文，并在首次出现时解释。
   - 密度：拆长段，删空话，保留影响执行的约束。
4. 完成复查：
   - 事实、数字、路径、字段、业务规则和决策保持一致。
   - 中英文混用都有明确必要。
   - Markdown 标题、列表、示例和段落便于阅读。
   - 如果本轮新增或修改了 Markdown，最终回复标明 `中文表达自检：已做`；本轮未新增或修改 Markdown 时标明 `中文表达自检：不适用`。

## 审校输出

用户要求审校、评估或给建议时，使用这个结构：

1. **总体判断**：用 2-4 句话说明主要问题和优先处理方向。
2. **高优先级问题**：只列会影响理解、评审或执行的问题。
3. **局部改写建议**：按“问题 -> 建议写法 -> 原因”说明。
4. **可替换片段**：必要时给可直接替换的 Markdown 片段。
5. **仍需确认**：列出无法仅靠文本确认的事实、数字、路径或业务口径。

批量审校多个 Markdown 文件时，先输出文件分组、统一术语规则、高风险文件、建议修改顺序，以及是否适合直接批量改写。

## 表达规则

- 普通概念优先用中文，例如“规则”“流程”“输入”“输出”“示例”“兜底”“当前情况”“建议”。
- 必要术语可以保留，例如 `Agent`、`MCP`、`CLI`、API 名称、模型名、路径、命令、函数名、字段名和产品名。
- 减少不必要的中英混搭。把“同步 current status”“给 recommendation”改成“同步当前情况”“给建议”。
- 替换生硬句式：
  - “基于……进行……”改成“根据……判断 / 处理 / 制定……”。
  - “通过……来实现……”改成“用……实现……”或更直接的动词短语。
  - “被用于……处理……”改成“用来处理……”。
- 把空泛黑话改成能看见动作和结果的说法。例如“形成闭环”如果实际意思是“处理到有明确结果”，就直接写后者。
- 面向业务读者时，先讲发生什么、带来什么结果，再补实现细节。
- 面向研发读者时，保留准确标识，并用中文解释术语对应的动作、约束或结果。

## Markdown 规则

- 标题要具体，大小要和内容匹配。
- 长段落优先拆成短段、列表或表格；拆分服务理解，形式服从内容。
- 多层嵌套只在能减少理解成本时使用。
- 示例要贴近对应规则，放在读者需要它的位置。
- 修改 Markdown 后，对新增或修改段落做中文表达自检，确认表达自然、术语必要、事实稳定。
- 写后自检是交付门禁，属于文件修改后的必要步骤；完成文件修改后，在最终回复里说明自检结果。
