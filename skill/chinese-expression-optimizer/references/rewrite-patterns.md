# 中文表达检查清单与改写模式

当任务需要更细的审校、具体改写示例或批量统一规则时，读取这份参考。

## 快速检查清单

- 结论是否先出现，长铺垫是否只保留必要背景。
- 每段是否只承担一个主要意思。
- 普通概念是否优先使用自然中文。
- 必要英文是否属于专有名词、产品名、协议名、命令、路径、字段名或代码符号。
- 字段名、路径、命令、文件名、代码符号是否用反引号标出。
- 字段名是否有中文说明，例如“检查项 `checks`”。
- 是否存在翻译腔、客服腔、营销腔、公文腔或空泛黑话。
- 标题、结论句和规则句是否以正向表达为主。
- 面向业务读者时，是否先讲动作和结果，再补技术名词。
- 面向研发读者时，是否保留必要技术名词，并补足中文解释。
- 事实、数字、路径、业务口径和项目结论是否保持一致。

## 术语处理

### 普通概念优先中文化

普通概念优先使用中文：

- `rule` -> 规则
- `input` -> 输入
- `output` -> 输出
- `example` -> 示例
- `workflow` -> 流程
- `recommendation` -> 建议
- `current status` -> 当前情况
- `fallback` -> 兜底机制

### 必要标识保留原文

英文或原始标识本身就是准确名称时保留：

- 产品或工具名称：`Cursor`、`Codex`、`GitLab`、`Moxt`
- 上下文已有共识的技术概念：`Agent`、`MCP`、`CLI`、`API`
- 命令和路径：`npm run dev`、`/api/v1/tasks`
- 字段名和代码符号：`checks`、`check_id`、`expected`

保留标识时，如果读者可能不熟悉，要配中文含义：

- 检查项 `checks`
- 检查项编号 `check_id`
- 预期结果 `expected`
- 命令行工具 `CLI`

## 改写模式

### 模式一：替换生硬翻译腔

问题：

```markdown
我们需要基于现有信息来进行下一步策略制定。
```

推荐写法：

```markdown
我们要根据现有信息判断下一步怎么做。
```

原因：中文里直接说动作更自然。

### 模式二：减少不必要的中英混用

问题：

```markdown
我会先同步 current status，再给 recommendation。
```

推荐写法：

```markdown
我会先同步当前情况，再给建议。
```

原因：这里的英文没有增加精度，反而打断阅读。

### 模式三：字段名准确，并配上中文含义

问题：

```markdown
用例中的 checks 字段只引用 check_id。
```

推荐写法：

```markdown
用例里的“检查项”只引用已定义的检查项编号，也就是 `checks` 和 `check_id`。
```

原因：非研发读者先看到含义，研发读者也能拿到准确字段名。

### 模式四：把抽象黑话改成可见动作

问题：

```markdown
主流程负责结构化事实预处理、触发被测链路、出站截获和硬兜底。
```

推荐写法：

```markdown
工具会先整理可核对的事实，触发被测流程，并在真实发送前拦截用户可见消息；遇到高风险动作时直接拦截或交给人工复核。
```

原因：改写后能看到具体发生什么，以及带来什么结果。

### 模式五：必要工程术语可以保留

问题：

```markdown
幂等、回注、干跑、硬兜底都要处理。
```

推荐写法：

```markdown
这段逻辑要支持重复执行不重复写入，也就是幂等；同时要能把结果写回原系统，支持只跑流程但不真实发送的干跑模式，并在异常时强制拦截。
```

原因：工程术语仍然保留，但读者能理解对应动作。

### 模式六：规则句优先使用正向表达

问题：

```markdown
本方案不是恢复旧链路，而是新增批量读取能力。
```

推荐写法：

```markdown
本方案采用批量读取能力，Agent 按需读取指定模块，工程侧负责写入和校验；旧的每轮全量读取链路只作为迁移边界说明。
```

原因：读者先看到目标动作，旧路径留作边界背景。

问题：

```markdown
不要把异常排查写进 PRD。
```

推荐写法：

```markdown
异常排查归入技术方案；PRD 保留会影响产品体验和验收的异常要求。
```

原因：改写后能看到归属和动作，而不是只看到禁止项。

## 按读者选择表达

### 业务、产品、运营、教研、测试读者

优先：

- 说明用户或流程会发生什么变化。
- 说明结果如何检查。
- 说明重要风险和边界。
- 先讲动作，再补技术名词。

表达取舍：

- 字段名配中文说明后再出现。
- 实现细节放在动作和结果之后。
- 长名词链拆成动作、对象和结果。

### 研发读者

优先：

- 保留准确标识、接口、状态、命令和约束。
- 补一句中文解释，说明标识为什么重要。
- 同一概念在文档中保持叫法一致。

表达取舍：

- 代码标识保持原文。
- 宽泛说法只作为解释，准确名称保持可见。
- 边界条件放在读者容易看到的位置。

### 管理层或评审材料

优先：

- 结论先行。
- 说明业务影响、需要的判断、风险和下一步动作。
- 用具体证据支撑判断。

表达取舍：

- 过程流水只保留能支撑判断的部分。
- 形容词服务证据和判断。
- 价值表述要对应可观察变化。

## 批量审校流程

审校多个 Markdown 文件时：

1. 按读者和用途给文件分组。
2. 先统一术语，再开始改写。
3. 标出需要事实、数字、路径或决策复核的高风险文件。
4. 先改风险最低的一组。
5. 复查改过的段落，确认事实稳定、术语一致。
6. 说明改了什么、保留了什么、哪些仍需人工确认。

## 稳定边界

- 表达优化以事实稳定为前提。
- 约束、风险和 `待确认` 项放在读者容易理解的位置。
- 实用工作文档保持可执行，表达优化服务理解和行动。
- 产品名、代码标识、路径、命令和字段名保留原文。
